# QQMessageManager

QQMessageManager 是一个基于 Python 桌面窗口的 QQ 消息统一管理客户端，用于通过 [NapCatQQ](https://github.com/NapNeko/NapCatQQ) 的 OneBot 正向 WebSocket 接口实时接收 QQ 私聊和群聊消息，并以接近 QQ 的会话列表 + 消息窗口形式展示和发送文本消息。

默认连接地址：

```text
ws://127.0.0.1:3001
```

端口和 Token 由用户在登录窗口中输入，Token 可以留空；程序会缓存上次成功点击连接时填写的登录配置，避免每次重复输入。

## 当前功能

- 登录页输入 NapCatQQ WebSocket 地址、端口与 Token
- 自动缓存上次输入的完整地址、Host、Port、Path、Token 和连接模式
- 默认使用 `ws://127.0.0.1:3001`
- 登录连接成功后，自动请求最近 20 个会话，并为每个会话加载最近 20 条历史消息
- 历史消息会按时间插入会话，不计入未读消息
- 通过 WebSocket 实时监听 NapCatQQ / OneBot 消息事件
- 自动区分群聊消息和私聊消息
- 自动调用 `get_group_info` 获取群名，并在会话列表和聊天标题中显示真实群名
- 自动补全私聊昵称/备注，私聊会话优先显示人名而不是 QQ 号
- 自动过滤图片消息段和 CQ 图片信息，避免在聊天窗口显示图片 URL 或图片详情（可选开启“允许读取图片”后，AI 可结合图片内容回复并显示缩略图）
- 新消息到达时，只有滚动条原本就在底部才会自动滚到底；正在翻历史时不会被打断
- 左侧 QQ 风格会话列表，显示群聊和私聊
- 支持右键会话置顶/取消置顶，置顶会话会排在列表顶部并使用浅黄色背景显示
- 置顶状态会缓存到当前系统用户的 Qt 配置中，下次启动自动恢复
- 支持全局 AI 代管配置，服务商支持 `Minimax-m3`、OpenAI、DeepSeek，以及自定义 OpenAI 兼容接口（可自定义 API 地址与模型名称）
- 支持在 AI 设置中选择本地 Skill，默认“无”，当前内置 `shuimen`，会读取 `qq_message_manager/skills/shuimen/SKILL.md` 并注入到 AI system prompt，用于约束机器人说话格式和表达风格
- 支持实际视觉输入：图片成功读取后会转换成标准 PNG/JPEG，并通过内部 `vision` Skill 要求视觉模型先看图再回复
- 支持仅在“被 @ + 明确要求画图”时启用内部图片生成 Skill；支持的当前模型会生成并通过 NapCat 发送图片，不支持时会 @ 请求者并明确说明当前模型不支持生成图片
- 支持记忆收到过的 mface/marketface 表情包，最多 50 个，超过后自动淘汰最少使用且最久未用的表情包
- 支持 AI 从已记忆表情包中选择一个追加发送，前端会先发送文字，再追加表情包
- 支持 AI 生成回复后按字数模拟打字延迟，较长回复不会秒回
- 支持点击“总结”按钮总结当前单个群聊或私聊，会按设置的时间区间和消息最大数量主动读取历史消息后交给 AI 总结
- 支持每个群聊/私聊单独开启 AI 代管，不影响其他未代管会话
- AI 代管支持自定义 Prompt、收到非自己发言后随机延迟发送、最近发言限制、上下文条数、被艾特随机延迟回复、禁止连续自说自话、AI 自主判断不发送等规则
- AI 配置、API Key、Skill 选择、表情包开关、规则配置和被代管会话列表都会缓存到当前系统用户的 Qt 配置中
- 右侧消息窗口展示当前会话的消息时间、发送者和内容
- 底部输入框发送文本消息，群聊使用 `send_group_msg`，私聊使用 `send_private_msg`
- 未读消息计数
- 自动重连与连接状态提示
- 支持常见 CQ 码/消息段的文本化展示，例如语音、表情、@、回复等

## 运行环境

- Python 3.10+
- NapCatQQ 已启动并开启 WebSocket 服务

## 安装

```bash
pip install -r requirements.txt
```

## 启动

```bash
python main.py
```

或：

```bash
python -m qq_message_manager
```

## NapCatQQ 连接说明

程序当前作为 WebSocket 客户端运行，所以 NapCatQQ 里应开启“WebSocket 服务端 / 正向 WS”，不要配置成“WebSocket 客户端 / 反向 WS”。

推荐在 NapCatQQ 网络配置中新建：

```text
类型：WebSocket 服务端 / 正向 WS
Host：127.0.0.1
Port：3001
Path：留空
messagePostFormat：array
reportSelfMessage：false
Token：没有鉴权需求时留空
启用：开启
```

程序启动后会打开登录窗口：

- 完整地址：默认 `ws://127.0.0.1:3001`
- Host：默认 `127.0.0.1`
- Port：默认 `3001`
- Path：默认留空；只有你在 NapCatQQ 里设置了自定义路径时才填写
- Token：填写 NapCatQQ 正向 WS 配置中的 token，没有则留空

如果你已经有完整地址，也可以直接在“完整 WebSocket 地址”中填写，例如：

```text
ws://127.0.0.1:3001
```

注意：NapCatQQ 的 WebUI 管理端口常见为 `6099`，不要把 WebUI 地址或调试地址当作本程序的消息接收地址。

## 使用说明

- 点击“连接”后，程序会把本次填写的完整地址、Host、Port、Path、Token 和连接模式保存到当前系统用户的 Qt 配置中；下次启动会自动填回登录窗口。
- 登录连接成功后，程序会先调用 `get_recent_contact` 获取最近会话，再对群聊调用 `get_group_msg_history`，对私聊调用 `get_friend_msg_history`，每个会话最多加载 20 条历史消息。
- 如果你的 NapCatQQ 版本不支持某个历史接口，程序会在连接日志中提示，实时收发消息仍可继续使用。
- 收到群消息后，程序会自动通过 `get_group_info` 请求群信息；请求成功后，左侧会话列表和右侧聊天标题会从“群聊 123456”更新成真实群名。
- 收到私聊或加载私聊历史时，程序会优先使用备注、昵称等信息显示私聊人名；必要时会调用 `get_stranger_info` 补全昵称。
- 未开启“允许读取图片（需要视觉模型）”时，图片消息段仍会被过滤，纯图片消息显示为“[图片消息已过滤]”；开启后程序会下载图片、裁剪明显空白、转换格式并作为真实多模态输入传给当前模型。当前模型或接口不接受图片时会在日志中明确报错，不会假装识图成功。
- 在左侧会话列表右键某个会话，选择“置顶会话”或“取消置顶”即可切换置顶状态；置顶会话会排在最上方并以浅黄色背景显示。
- 在发送栏旁边点击“AI设置”可以配置全局 AI 规则；支持 `Minimax-m3`、OpenAI、DeepSeek 及自定义 OpenAI 兼容服务商，需要填写对应 API Key（自定义需额外填写 API 地址与模型名称）。
- AI 设置里的 “Skill” 默认是“无”；选择 `shuimen` 后，程序会读取 `qq_message_manager/skills/shuimen/SKILL.md`，把它作为本地 Skill 注入给 AI，用来约束机器人说话格式、口癖、节奏、互动规则和角色表达。
- 图片生成 Skill 不需要在下拉框中选择。只有会话已开启 AI 代管、消息实际 @ 机器人，并明确包含“生成图片、画一张图、制作海报”等请求时才会触发。没有 @ 时只按普通聊天处理，不会自动出图。
- 图片生成使用当前配置的模型判断能力：OpenAI `gpt-5` 系列及更新主模型会尝试使用 Responses API 图片生成工具；`gpt-image-*`、`dall-e-*` 或明确的自定义 OpenAI 兼容图片模型会尝试使用 Images API。MiniMax-M3、DeepSeek 等当前文本模型会直接回复请求者“当前模型不支持生成图片”。接口运行时拒绝图片工具时也会按不支持处理。
- 图片生成成功后，文件会临时保存到系统临时目录，并以 OneBot `image` 消息段发送到当前群聊或私聊；生成失败不会发送伪造链接，也不会向聊天对象暴露 API Key、内部路径或完整接口错误。
- 开启“记忆收到的表情包（最多 50 个）”后，程序会从收到的 `mface` 或疑似 marketface 图片消息中提取 `summary`、`emoji_id`、`emoji_package_id`、`key`、`url/path/file_id/file_unique` 等字段写入本地表情包记忆库。已有表情包不会重复新增。
- 表情包记忆库最多保留 50 个；超过 50 个时，程序会优先删除使用次数最低的表情包，次数相同则删除最久未使用/最早创建的表情包。
- 开启“允许 AI 使用已记忆表情包”后，AI 会收到一个精简表情包列表，只能从列表中选择 ID，并可在回复末尾输出 `<STICKER:id>`。前端会移除这个标识，先发送文字，再追加发送对应表情包。
- 如果 AI 选中的表情包不存在、缺少可发送字段或发送开关关闭，程序会只发送文字并在日志中提示，不会把 `<STICKER:id>` 发到 QQ。
- 开启“根据文本长度延迟发送”后，AI 生成回复会根据正文长度模拟打字延迟；18 字以内基本不额外延迟，超过后按约每字 110ms 加随机抖动计算，最长约 18 秒。该开关默认关闭。
- 点击发送栏旁的“总结”按钮，可以总结当前单个群聊或私聊。弹窗里可设置开始时间、结束时间和消息最大数量；默认不限制时间区间，消息最大数量为 200。程序会向 NapCat 主动请求该会话历史消息，再结合窗口内已有消息去重后调用 AI 总结，而不是只总结当前窗口已经加载的消息。
- 总结结果会以弹窗显示，包含总览、主要话题、结论/待办、氛围与关系变化、需要回看原文的点。
- 在某个会话里勾选“AI代管”后，AI 只会代管该会话；其他未勾选的群聊/私聊不会自动回复。
- AI 代管规则包括：全局 Prompt、本地 Skill、收到非自己发言后等待 x~x 秒随机回复、最近 x 秒无人发言时不发、参考最近 x 条消息、被艾特后 x~x 秒随机回复、上一条是自己时不发、允许 AI 自主判断本次不发送。
- “收到新消息后自动回复”开启后，每次该会话收到新的非自己消息，都会重置一次随机延迟计时器；被 @ 的消息会优先走 @ 回复，不会同时排普通回复。
- 开启“允许 AI 判断本次无需回复”后，AI 可以返回内部标记 `__NO_REPLY__`；程序会把它当作空回复处理，不会把该标记发到 QQ。
- 选择左侧任意私聊或群聊，在底部输入框输入文字并按 Enter 或点击“发送”即可发送文本消息。
- 建议 NapCatQQ 的 `reportSelfMessage` 保持 `false`，因为程序会在本地立即显示自己发出的消息，避免服务端再回推自己的消息导致重复显示。

## 说明

本项目当前支持消息接收、最近历史消息加载、群名/私聊名展示、图片显示与视觉输入、会话置顶、AI 代管、Skill 选择、@ 触发图片生成、表情包记忆/发送、AI 打字延迟、聊天总结和文本消息发送。后续可以扩展：

- 消息搜索
- 本地消息持久化
- 多账号管理
- 图片/文件手动发送
- 更多 AI 服务商和独立图片模型配置
- 群聊过滤与更多置顶管理
- 托盘常驻提醒
