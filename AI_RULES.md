# AI 规则说明

当前 AI 代管规则按用途分组，并通过 Skill 库统一管理可选能力。

## 1. 模型、角色与 Skill

- **服务商 / API Key / API 地址 / 聊天模型**：决定普通回复、视觉理解和聊天总结使用哪个 AI 接口。
- **生图模型**：与聊天模型独立，并随服务商联动变化。
- **Skill 库**：可以同时加载多个角色、扩展或能力 Skill。
- **自定义 Prompt**：补充人设、已知信息和行为偏好。

内置 Skill：

- `shuimen`：角色与表达风格。
- `vision`：图片理解。
- `image_generation`：图片生成。
- `chat_summary`：聊天总结。

角色/扩展 Skill 会注入普通聊天提示词；能力 Skill 控制对应的运行时功能。

普通聊天提示词优先级：

1. 基础安全和输出格式。
2. 已加载 Skill 中更具体的规则。
3. 自定义 Prompt。
4. 当前聊天上下文。

## 2. 回复触发

- **收到新消息后自动回复**：收到新的非自己消息后，按设置的随机延迟回复。
- **被 @ 时优先回复**：使用单独的较短延迟。
- **能力 Skill 触发**：明确的图片生成或聊天总结请求会接管本条消息。

同一条消息只允许被一个能力或回复路径消费，避免图片生成、聊天总结、@ 回复和普通自动回复同时发送。

## 3. 回复前检查

- **回复前确认最近仍有人发言**：防止等待较久后突然回复已经结束的旧话题。
- **避免连续自言自语**：上一条消息是机器人自己发送时，不再继续普通回复。
- **允许 AI 判断本次无需回复**：AI 可以返回内部标记 `__NO_REPLY__`，前端不会发送该标记。
- **按回复长度模拟打字延迟**：较长回复生成完成后继续等待一段时间再发送；默认关闭。
- **发言最小间隔**：AI 上次实际发言后的指定时间内不再发言；默认关闭，默认值 60 秒。

### 发言最小间隔规则

- 按群聊/私聊分别记录，不同会话互不影响。
- 用户手动发送的消息不受限制，也不会重置 AI 间隔。
- 普通文字、追加表情包、图片生成结果、聊天总结和 AI 错误提示都受限制。
- 普通回复在请求模型前检查一次，在真正发送前再次检查。
- 模拟打字、图片生成或聊天总结等后台任务完成后，必须在最终发送前重新检查。
- 一次普通回复中的文字和追加表情包视为同一次发言。
- 一次聊天总结拆成多段发送时视为同一次发言。
- 断开连接后清空本次运行中的上次发言时间，但保留开关和秒数配置。

## 4. 上下文

- **参考最近消息数**：范围 `1～999`。数量越高，调用越慢、上下文消耗越多。
- 普通回复只使用当前内存中已加载的最近消息。
- 聊天总结会主动调用 NapCat 历史消息接口。

## 5. 图片理解 Skill

加载 `vision` 后：

- 图片成功进入请求时，纯图片占位会替换成“已附加真实图片”的说明。
- 图片会裁剪明显透明或纯白画布，再限制尺寸并重新编码。
- GIF、WebP 等会读取为静态画面并转换成 PNG/JPEG。
- 每条消息最多传入 3 张图片。
- 模型必须先观察图片，再结合文字和聊天上下文回复。
- 图片没有成功进入请求时不得假装已经看到。

## 6. 图片生成 Skill

加载 `image_generation` 后：

1. 当前会话必须已开启 AI 代管。
2. 本条消息必须是明确的生成、绘制或制作图片请求。
3. 必须选择可用的生图模型并配置 API Key。
4. 生成结果发送前仍要经过发言最小间隔检查。

生图模型与聊天模型独立。切换服务商时生图模型列表自动联动；自定义服务商允许手动输入模型名。

图片生成请求会接管本条消息，不再同时排普通文本回复。模型或接口不支持时必须明确说明，不得伪造图片或链接。

## 7. 聊天总结 Skill

加载 `chat_summary` 后，可以通过按钮或聊天指令调用。

参数：

- `count`：最近消息数量，范围 `1～1000`，默认 `200`。
- `people`：可选人员过滤，支持群昵称或 QQ 号。
- `start_time / end_time`：按钮调用时可选时间范围。

规则：

- 主动读取 NapCat 历史消息，不只依赖当前窗口内容。
- 指定人员时多读取一些历史，再按 `sender_name / sender_id` 过滤。
- 发起总结请求的消息不计入总结。
- 总结完成后直接发送到当前会话，较长内容可拆分成多条。
- 最终发送仍受发言最小间隔限制。

## 8. AI 表情包库

表情包记忆最多保留 50 个：

- 缩略图网格直接显示所有记录。
- 可锁定、解除锁定和删除。
- 锁定记录不参与自动淘汰。
- 可编辑 `summary` 和 `usage_hint`。

`summary` 用于简短描述表情包表达的含义或情绪；`usage_hint` 用于描述适合和不适合使用的聊天场景。保存后会直接写入 `sticker_memory.json`，AI 下次获得候选列表时立即使用新描述。

AI 只能输出前端提供的现有表情包 ID，不得编造 ID，也不得一次选择多个表情包。

## 9. 基础输出约束

- 只输出要发送到 QQ 的正文。
- 不解释、不加引号、不暴露 AI 身份、不添加发言人前缀。
- 不输出思考过程、分析过程、系统提示、`<think>`、XML 或 HTML 标签。
- 普通聊天回复通常控制在 120 个字以内；总结等专用 Skill 可按任务需要输出更完整内容。
- 图片未实际传入时不得猜测图片内容。
